Build a production-grade Documentation Validation Agent inside the existing Dynasties Replit project. This must integrate with the current architecture, current shipment/document models, existing AI layer, existing compliance agent, and the current UI. Do NOT create a separate prototype, toy app, or parallel system.

WORKING STYLE
- Plan first before coding.
- Inspect the existing repo and reuse current services, shared types, schemas, validators, and UI patterns.
- Do not introduce duplicate AI clients, duplicate document models, or duplicate workflow logic.
- Build incrementally with checkpoints.
- Keep all code production-oriented, typed, and integrated.
- Do not overwrite working code.
- Prefer narrow composable services over monolithic files.
- Reuse the existing Anthropic integration if present.
- Show progress as: repo analysis -> implementation plan -> files to modify -> code changes -> tests -> manual verification -> open issues.

PRIMARY GOAL
Implement a real Documentation Validation Agent that evaluates whether shipment-related documents are complete, internally consistent, and operationally usable. It must detect missing required documents, missing required fields, field inconsistencies across documents, suspicious document content, and readiness for downstream workflow. It must return a structured decision that visibly affects workflow.

THIS IS NOT
- not a generic chatbot
- not a fake “AI validator”
- not a separate dashboard with mock data
- not only OCR/extraction
- not only document generation
- not a passive report with no workflow consequence

THIS IS
- one narrow, real, outcome-affecting agent
- integrated into existing shipment/document workflow
- able to operate on real shipment and document data
- able to return structured validation output
- able to be shown in a client demo
- extensible later

BUSINESS CONTEXT
Dynasties is being positioned as governed trade execution infrastructure. Documentation quality is one of the clearest sources of operational failure and manual review in freight workflows. This agent should make the product feel like real operational infrastructure, not AI decoration.

EXISTING SYSTEM CONTEXT
Before coding, inspect and explicitly map integration points with the existing:
- document-extraction service
- shipment-construction service
- document-generation service
- compliance agent / compliance screening results
- risk scoring
- current shipment detail page / workflow UI
- existing database tables for ingested documents, shipments, events, and any extracted_data or metadata
- existing Anthropic integration wrapper

The audit shows the platform already has document extraction, shipment construction, document generation, billing, pricing, exception management, and an Anthropic integration wrapper, so this agent must fit inside those existing rails rather than introducing a second architecture. It also notes the agent pattern should remain advisory with deterministic validation and structured output. AI PROVIDER INTEGRATION (CRITICAL)
1. Inspect the codebase for an existing AI provider.
   - If an AI service already exists:
     - Reuse it.
     - Do NOT create a second AI client.
     - Route all AI calls through the existing abstraction.
2. If no reusable provider abstraction exists:
   - Implement a clean wrapper around Anthropic using the existing project convention.
3. Fix existing issues if present:
   - prevent crashes if API key is missing
   - use lazy initialization
   - add error handling and fallbacks
4. LLM usage rules:
   - input must be structured
   - output must be structured JSON
   - low temperature / deterministic settings
   - used for ambiguity resolution, summarization, and explanation
   - NOT used as the sole source of truth for hard validation rules
5. System must still work without AI:
   - deterministic validation runs first
   - AI enhances reasoning and explains inconsistencies
6. If there is any Anthropic env mismatch in the current project, align the new code with the existing, working project conventions instead of creating another variant.

CORE FUNCTION
Given a shipment and its associated ingested/generated documents, the Documentation Validation Agent must:
1. Inspect the available document set for the shipment.
2. Determine whether the shipment has the required document types for the current workflow stage.
3. Validate whether required extracted fields are present within each document.
4. Detect inconsistencies across documents.
5. Flag suspicious or incomplete documentation that should block or delay workflow.
6. Return a structured result:
   - status: READY | REVIEW | BLOCKED
   - readinessLevel: COMPLETE | PARTIAL | INSUFFICIENT
   - missingDocuments: array
   - missingFields: array
   - inconsistencies: array
   - suspiciousFindings: array
   - recommendedActions: array
   - reasoningSummary
   - reviewedAt
7. Persist the result and show it in the UI.
8. Affect workflow outcome.

SUPPORTED DOCUMENT TYPES
Adapt to whatever document model exists in the codebase, but support validation concepts for typical freight docs such as:
- commercial invoice
- packing list
- bill of lading / HBL / MBL
- arrival notice
- customs-related documents
- insurance documents if present
- shipment summary or generated internal document if already part of system

Do not hardcode assumptions that every shipment requires every document type. Required docs should depend on shipment type, workflow stage, and available metadata/rules.

EXAMPLE VALIDATION TASKS
The agent should be able to detect:
- missing commercial invoice
- missing packing list
- missing bill of lading
- missing consignee / shipper / carrier field
- missing destination/origin
- missing invoice amount / currency where required
- missing package count / weight where relevant
- mismatched consignee name across docs
- mismatched commodity description across docs
- mismatched invoice value across docs
- mismatched ports / destination across docs
- missing or inconsistent document dates
- unsupported/low-confidence extracted fields
- generated documents that cannot proceed because source data is incomplete

OUTPUT CONTRACT
Create a strongly typed output schema, e.g.

DocumentationValidationResult = {
  shipmentId: string;
  status: "READY" | "REVIEW" | "BLOCKED";
  readinessLevel: "COMPLETE" | "PARTIAL" | "INSUFFICIENT";
  missingDocuments: Array<{
    code: string;
    label: string;
    reason: string;
    severity: "WARNING" | "CRITICAL";
  }>;
  missingFields: Array<{
    documentType?: string | null;
    field: string;
    severity: "WARNING" | "CRITICAL";
    detail: string;
  }>;
  inconsistencies: Array<{
    code: string;
    field: string;
    values: string[];
    severity: "WARNING" | "CRITICAL";
    detail: string;
  }>;
  suspiciousFindings: Array<{
    code: string;
    title: string;
    detail: string;
    severity: "WARNING" | "CRITICAL";
  }>;
  recommendedActions: string[];
  reasoningSummary: string;
  sourceDocuments: Array<{
    documentId: string;
    documentType?: string | null;
    filename?: string | null;
    validationState: "VALID" | "INCOMPLETE" | "CONFLICTED" | "UNKNOWN";
  }>;
  reviewedAt: string;
};

ARCHITECTURE
Create a dedicated service/module, such as:
- services/document-validation/

Separate:
- document inventory / normalization
- required-document rules
- required-field validation
- cross-document consistency checks
- AI reasoning layer
- output builder

If there is an existing service pattern like agent.ts, validator.ts, index.ts in other service modules, follow that exact pattern so this agent feels native to the codebase.

REQUIRED DOCUMENT RULES
Implement a deterministic rules layer first.
Rules should be configurable and extensible, not hardcoded in route handlers.

Examples:
- Shipment in DRAFT/PENDING_REVIEW requires at least core operational docs to continue
- Missing invoice and packing list => REVIEW or BLOCKED depending on shipment stage
- Missing bill of lading at a downstream stage => REVIEW or BLOCKED
- Missing values required for pricing, billing, compliance, or claims should be surfaced
- If extraction confidence or extracted field availability exists in the current model, use it

CROSS-DOCUMENT CONSISTENCY
Implement checks for:
- shipper name consistency
- consignee name consistency
- notify party consistency if present
- commodity description consistency
- origin/destination consistency
- invoice value consistency
- package count / weight consistency where available
- dates and reference numbers where meaningful

AI LAYER (SECOND PASS)
Use AI only to:
- summarize the validation result
- explain likely causes of inconsistency
- classify ambiguous conflicts when deterministic logic is insufficient
- produce concise recommended actions

Never let AI overrule deterministic missing-document or missing-field rules.
