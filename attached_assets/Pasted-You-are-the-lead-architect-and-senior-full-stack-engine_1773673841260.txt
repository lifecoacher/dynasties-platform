PHASE 1 — PRODUCTION HARDENING

Replace any in-memory / EventEmitter queue implementation with a durable queue architecture suitable for production. Use the already-provisioned AWS-style queue architecture pattern.

Implement:
- durable async job publishing
- worker consumers
- retry logic
- dead-letter queue support
- idempotency protection for duplicate jobs

Wrap all multi-step database write operations in transactions.

Replace any local filesystem upload/storage behavior with cloud object storage abstraction suitable for containerized deployment.

Harden multi-tenant boundaries:
- fix all cross-tenant admin exposure risks
- ensure tenant/company scoping is enforced everywhere
- restrict admin actions to the tenant unless a true super-admin role is explicitly introduced

Add:
- runtime request validation
- rate limiting on sensitive endpoints
- structured logging
- global error handling
- test coverage for core pipeline flows

Implement a normalized canonical data model and migrations for the following entities if they do not already exist in sufficient form:

- shipper

- freight_forwarder
- carrier
- vessel
- port
- trade_lane
- commodity_hs_code

Create relationship/edge tables or equivalent relational structures so Dynasties can represent a trade intelligence graph.

PHASE 3 — DECISION ENGINE

Create a new backend service/module called something like decision-engine.

Its job is to consume outputs from:
- shipment construction
- compliance screening
- risk intelligence
- insurance
- pricing
- trade-lane intelligence
- exceptions

For each shipment, generate structured recommendations such as:
- carrier switch
- route adjustment
- insurance adjustment
- compliance escalation
- delay warning
- pricing/margin warning
- document correction urgency

Every recommendation must include:
- recommendation id
- shipment id
- tenant/company id
- recommendation type
- title
- explanation
- machine-readable reason codes
- confidence score
- urgency
- expected delay impact
- expected margin impact
- expected risk reduction
- recommended action
- created at
- status

Recommendation generation should be deterministic-first where possible, with AI-generated explanation layered on top only where needed.

Expose recommendations through API endpoints and wire them into the current shipment lifecycle.

Implement full recommendation outcome tracking.

Create a recommendation_outcomes model/table and logic to record:
- recommendation shown to operator
- accepted
- modified
- rejected
- implemented

- actual claim result
- actual cost delta
- actual margin delta

- which recommendations improve outcomes

- which decision patterns create better results

Important:
Do not build model retraining yet unless it is already easy to support.
Instead, build the data exhaust and analytics foundation required for future learning.
